## 論点 D-1: @spec / @stack アンカーは必要か

### ユーザー見解（起点）
> @stack のうまみってなんなんだ？ → ぶっちゃけあんまりいらない。／@spec も DDD においてそこまで厳密にする必要があるのか。／リンクの管理自体がうまく回らない気がする。ここが陳腐化しそう。

### AI 見解（敵対的自己検証の結果）
- **@stack**: コードの import 文が既に技術を語る＝アンカーは重複情報。**廃止**。tech-stack ドキュメント（能力→実装の登録簿）は残す。
- **@spec**: 正典配置（architecture 規約）が既にリンク＝`uc-place-order` → `application/usecases/place_order.py` は**パスから導出可能**。アンカーが要るのは共有コード等の例外だけ——だが「例外だけ手書き」は一番崩れやすい規律で、**守りが濃く必要な場所ほど防御が弱い**逆転構造。
- リンク機構の設計連鎖（全必須→弱める→配置＋例外→ゲートで守る）は**未検証の理屈の積み上げ**だった＝ユーザーの「しっくりこない」の正体。

### 合意決定
**カスタムアンカー全廃（@spec / @stack とも）。リンクは「管理」するものではなく「計算」するもの**（正典配置＋検索＋機械導出）。spec↔code 対応の計算精度は Stage B 設計時に保守シミュレーション実験で検証（理屈で決着させない）。

## 論点 D-2: DDD に詳細設計書は要るか

### ユーザー見解
> DDD の「コードに設計が宿る」において、構造的レンダリングまでやらないと再現性を担保できないのか？ 昔は AI がないので厳密な詳細設計が必要だったかもしれないが今日日いらないのでは。詳細設計の本来の役割に立ち返るべき。

### AI 見解（詳細設計書の4役割と AI 時代の帰結）
| 旧役割 | AI 時代 |
|---|---|
| ① 分業の伝達（設計者→コーダー命令書） | 分業は戻る（人=意図/AI=コーダー）が、**運ぶのは詳細でなく意図**（AI は詳細を自分で埋める）＝spec と規約まで |
| ② 事前検証（実装が高価だった） | **事後ゲートへ逆転**（実装が安価＝書いて検証すればいい） |
| ③ 記憶の永続化 | 意図と決定だけ永続化（詳細はコードを読めば分かる） |
| ④ 再現性 | 同一コードの再現は不要。**振る舞いの再現（シナリオ）と発見可能性**で足りる |

DDD（2003）は「分業をなくす」ことで①を解体した（設計はコードに宿る）。AI は「大量のコードを人が読めない」という DDD の唯一の弱点を埋める（読み手が AI）＝**AI 時代にこそ DDD の立場は純粋に貫ける**。

### 合意決定
**詳細設計は「成果物」から「クエリ結果」になる。** 永続化するのは 意図（spec）／制約（規約）／振る舞い検証（シナリオ）の3つだけ。詳細はコードに宿り、説明が要るときは AI/engine がその場で導出する（保存しない）。API 仕様・DB スキーマ文書・インフラ文書は**境界契約/投影＝コードから機械生成**（手書き並行維持を構造的に禁止）。画面設計だけは意図＝PresentationSpec（別家族・将来）。

## 論点 D-3: トークン経済と「導出物=キャッシュ」

### ユーザー見解（D-2 への重要な反論）
> 「再現性が不要」は気になる。**似たようなコードの量産は AI 時代の新たな問題**で、膨大なトークンを使う。いいモデルと無限トークンがあればこのツール自体不要（見つかるまで参照し続ければいい）。**無駄なものを作らないことと、高いドキュメント検索性は、トークン量の観点から非常に大事**。

### AI 見解（修正）
初版の「読むとき AI が計算すればいい」は**正しさの理論としては真・経済としては偽**（毎セッション同じ導出を払う／探し損ね→類似コード量産→肥大→さらに高くなる悪循環）。正解は第3の道:

| 導出物の扱い | 陳腐化 | トークン |
|---|---|---|
| 手で維持（旧詳細設計書） | 腐る | 安い |
| 保存せず毎回 AI が計算 | 腐らない | **毎回高額** |
| **保存 OK・機械が決定的に再生成（AI 0）** | **腐らない** | **ほぼゼロ** |

**陳腐化とは「キャッシュを人が手で更新する」ことの別名。** 「保存するな」ではなく「**手で維持するな**」が正しい規範。これはハーネス原則（機械=engine／AI=判断だけ）のトークン経済からの再導出＝独立検証。

### 合意決定
- **導出物＝キャッシュ**（機械が決定的・無償で再生成できるものだけ保存してよい）。
- **spec＝検索可能な意図カタログ**＝重複量産の第一防壁（実装前にカタログを引く）＋理解のキャッシュ（50行の spec を読み 5000 行のコードを読まない）。
- **has-udd の存在意義の最終言語化＝「AI 開発のトークン経済を成立させるハーネス」**。

## 論点 D-4: コードの検索性（DocComment 動的インデックス）

### ユーザー見解（発案）
> 全ソースが決められた DocComment 構造を持つなら、実装の中身を見なくても当たりがつけられる。document.json が無くても、クエリが動的にインデックスビューを作る仕組みがあれば、前読み込みなしで概要を掴める。

### AI 見解＋PoC 検証（2026-07・成功）
- Python 標準 `ast` のみ（依存ゼロ・AI 0・決定的）で docstring を機械抽出 → JSON ビュー `{path, kind, name, summary}`。
- **実測: src/has_udd 28 ファイル / 1,293 行 → ビュー約 750 トークン（1/7 圧縮）**・「レンダリング関連どこ？」の当たり検索一発・本体未読。
- **距離の原理: 記述の陳腐化リスクは記述対象との距離に比例**。同ファイル docstring＝距離ゼロ＝同じ diff・同じレビュー＝腐りにくい（「実装準拠の書き方だから陳腐化しない」）。
- 多言語: ビュー契約は言語中立・抽出器は言語別 outbound adapter（Python=ast / TS=tree-sitter / …）。各言語のネイティブ慣習を借りる＝新形式を発明しない。

### 合意決定
**code_scan を query engine の操作として実装**（ES-3 の解）。インデックスは保存しない（読み取り時計算＝documents の _index と同型）。先行実装は Python 抽出器のみ。

## 論点 D-5: DocComment の形式

### ユーザー見解
> spec タグはいらなくなったのでは（stack タグも）。**一般的にドックコメントで記載できる表現を活用してちゃんと書くべき**。特に**サマリー部分はそのまま x-prompt-query と言っても過言ではない**。

### AI 見解
- カスタムタグ全廃の帰結＝**ソースコードに has-udd 固有記法がゼロ**（完全標準準拠・IDE/Sphinx 互換・侵襲ゼロ＝ハーネス原則の理想形）。
- **要約行（1行目）＝x-prompt-query 相当**: インデックスビュー経由で読む AI への読み方ガイド。schema の x-prompt-query と同じ品質基準で書く。
- 粒度の判定: 言語標準スタイル（Python=Google スタイル）が最善。残る検証点（①要約の質は機械検査不能 ②spec↔code 対応精度）は**どの粒度でも残る問題領域固有のもの**であり、規約でなくサンプル（①）と Stage B 実験（②）が持ち場。

### 合意決定
**DocComment 規約＝「その言語の標準スタイルでちゃんと書く」＋要約行の品質基準（x-prompt-query 相当）。カスタムタグなし。** 意味を書く（型の繰り返し禁止）・公開要素必須。coding-standard の DocComment ブロックとして確定（実物=`docs/design/coding-render-image-coding-standard.md`）。

## 論点 D-6: gen-gap マーカーの存廃

### AI 提案
`# has-udd:impl-start/end`（現コード69箇所）も同原理で**廃止**すべき: (a) has-udd 固有記法（「ソースに独自記法ゼロ」に反する）(b) 保護対象が消えた（コードは AI が author するもので、機械が再生成して上書きするものではない。scaffold が生成するのは document.json でありコードではない）。

### 合意決定
**廃止。** D-1〜D-5 と同じ原理（has-udd 固有記法をソースから排除・保護対象なきマーカーは不要）を適用。既存69箇所は @spec:/@stack: タグ削除と同じタイミング（Stage B 前後）でまとめて除去する。
