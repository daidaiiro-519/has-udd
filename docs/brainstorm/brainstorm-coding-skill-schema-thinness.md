# ブレインストーミング: CodingSchemaとSkillSchemaの内容が「弱い」という直感の正体

**目的:** 「CodingSchemaとSkillSchemaの内容が少し弱いかも」というユーザーの直感の正体を特定し、深めるべき箇所と手段（schema拡張か運用かブレスト継続か）を合意する。
**モード:** 問題解決

---

## 問題の再定義

CodingSchema・SkillSchemaの一部ブロックが「ルールや手順は書けているが、実際どう適用すればいいかが伝わりにくい」せいで、AIがdocumentを読んでも判断に迷う可能性がある。

## 原因候補

- 表面的原因: 一部ブロックに具体例（コードサンプル・実行例）フィールドが無い
- 表面的原因: 同じCodingSchema傘下でもブロックごとの厚みにばらつきがある（test-standardだけ突出）
- 根本原因候補: 「例を持たせる」という設計判断が最初から全ブロックに一貫適用されず、後付けで個別に足された（DocstringBlockのexampleは局所対応で終わった）
- 根本原因候補: 厚みの差は schema の欠陥ではなく、実際にそのブロックを使って意思決定を重ねた量の差（test-standardは今日1日で大量の決定が乗った）

---

## 論点 1: CodingSchemaの「ルールだけでコード例が無い」状態は本当に問題か？

### AI 初期見解
**見解:** 問題である。ただし全ブロック一律にではなく、「コードの形」を規定するブロック（Naming/Style/CodingRules/ArchitectureRules）に限って本当に困る。
**根拠:**
- DocstringBlockだけexampleを持つのは意図的な設計ではなく、docstring構造が特に複雑でprose(level+rule)だけでは表現しきれなかったための局所対応に見える（他ブロックへの横展開が漏れた）
- 「動く最小サンプルをexampleで配る」（project-coding-schema.mdに記録済みの方針）という既存合意がある以上、Naming/Style/CodingRulesがそれに従っていないのは方針とdocumentの間のドリフトそのもの
- ConceptPlacementの「pattern」フィールドは一行proseで疑似的にexampleの役割を担おうとしているが、実コード片ではないため曖昧さが残る（例: 「不変（frozen dataclass）」だけでは実際の書き方が一意に決まらない）

### ユーザー見解
> コード例は陳腐化の入り口だと思うが？

（続けて）
> いや、この部分の思想としては、スキーマで定義された項目が必ずドックストリングに書かれることは不変です。なぜならこれは言語などで書き方が決まってるからです。意味的推論まで機械的にやってほしいと言ってるわけではないです。私はその言語によって変わらない不変的なドックストリングの書き方があるので、そこは構造的に担保しようよというためのスキーマです。そしてこれを検知するのがreconcileだったはずですよ？コードスキャンでドックストリングのドックストリングDTOをかまして、それに対して検証することでソースコードのドックストリングが不足していないかどうかを機械的に検知できるはずです。これをコード例ということで載せた場合はそのコード例をみてAIが構造推論し記載する、そうするとコンテキスト次第で抜け漏れなどが発生する、それが今回の事象だと思う。

### AI 再考見解
**見解:** さらに訂正する。問題は「静的コード片の埋め込みが陳腐化する」ことでもなく、**`uc-lint-docstring`が本来担保すべき構造チェックを実際にはやっていない**ことだった。docstringの構造（Args/Returns/Raisesの要否）は言語仕様で決まる不変のルールであり、AIが例を見て毎回構造推論する話ではない。reconcile（`uc-lint-docstring`）がDocstringSchema DTOに対して機械検証すべきものであり、それが正しく機能していれば`example`という補助輪自体が要らない。
**根拠（実データで検証済み）:**
- `PydoclintLinter`アダプタは`--skip-checking-short-docstrings`を明示的に指定しておらず、pydoclintの既定値`True`のまま動いている。この既定値のせいで、一行docstring（要約のみ）は**構造チェックそのものがスキップされる**——`render_parts()`が一行docstringのままARGS_MISMATCH等を一切検知されなかったのはこのため
- 実際に`--skip-checking-short-docstrings=False`を付けて`part_renderer.py`に対して実行すると、`render_parts`（公開関数）が`DOC101`(Docstring contains fewer arguments than in function signature)・`DOC201`(does not have a return section)で正しく検知される。つまりreconcileの機構自体は機能する余地があり、単に**フラグ設定が甘くて発動していなかった**だけ
- 現在の`LintDocstringEngine`/`_parse_output`は、pydoclintが出す複数の診断コードのうち`DOC103`（ARGS_MISMATCH）だけを抽出し、他は全て握りつぶしている。DOC101（Args不足）・DOC201（Returns不足）等を無視している限り、たとえフラグを直しても検知結果に反映されない
- ただし全関数に無差別適用すると、private関数（`_seq_token`等）まで大量に引っかかる。これは既存のcoding-standard document自身の規約「公開要素は必須・privateは任意」と整合させる必要があり、`--skip-checking-private-functions`（既定`False`）を適切に設定するか、`PythonAstSourceScanner`側のelementKind/公開判定と組み合わせてフィルタする必要がある

### 合意決定
**決定:** CodingSchemaのexampleフィールドを増減する話ではなく、**`uc-lint-docstring`（PydoclintLinter+LintDocstringEngine）の検知範囲を拡張し、Args/Returns/Raises節の欠落を機械的に検知できるようにする**方向で合意。DocstringBlock.exampleは残すか削除するかは別途判断するが、いずれにせよ「AIが例から構造推論する」ことに頼らず、reconcileが実際に強制する状態を先に作る。
**理由:** ユーザーの指摘通り、docstring構造は言語仕様で決まる不変のルールであり、意味的推論ではなく機械検証すべきもの。実データ検証で「reconcileが実際には機能していなかった」ことが確定した以上、そこを直すのが正しい優先順位。
**次のアクション:** `uc-lint-docstring`のSpecに新しい違反コード（例: MISSING_ARGS_SECTION/MISSING_RETURNS_SECTION等）を追加し、`--skip-checking-short-docstrings=False`＋`--skip-checking-private-functions`の適切な設定＋`_parse_output`の対応コード拡張をTDDで実装する。

## 論点 2: 陳腐化を避けつつ「実際どう書けばいいか」を伝えるには、どんな形が良いか

（当初「schemaレベルでexampleフィールドを追加すべきか」という論点だったが、論点1の結論を受けて「静的コード埋め込み以外の手段」を検討する論点に組み替える）

### AI 初期見解
**見解:** 静的コード文字列の埋め込みではなく、**実コードへのポインタ（ファイルパス＋シンボル名）**を持たせるべき。文字列そのものではなく「どこを見ればよいか」だけを持たせれば、陳腐化してもコストが桁違いに小さい。
**根拠:**
- ポインタが古くなる（ファイルが移動・関数名変更）ことと、コード本文が古くなる（シグネチャやdocstringの中身がズレる）ことは同じ「陳腐化」でも被害が違う。前者は`uc-scan-source-code`で対象シンボルの実在を機械的に検証でき（既存engineの転用で済む）、かつ検知されれば「参照が壊れている」という分かりやすいエラーになる。後者（文字列比較）は意味的な食い違いを機械検知しにくく、気づかれないまま嘘をつき続ける
- 実コードは既に`uc-scan-source-code`/`uc-lint-docstring`で構造化スキャン・規約適合検証の対象になっている。「正しい書き方の実例」を知りたければ、実際に規約を満たしている実コードそのものを指させばよく、二重に保持する理由が無い
- これは「詳細設計=クエリ結果=キャッシュ・コードから導出できるものは複製しない」という既存の憲法級原則をexampleフィールドにもそのまま適用しているだけで、新しい原則を発明していない

### ユーザー見解
> （論点1のやり取りにより、この論点は前提が変わった）

### AI 再考見解
**見解:** この論点自体が不要になった。「exampleをどう陳腐化しないよう持たせるか」という問い自体が、exampleを持たせることを前提にしていたが、論点1の結論は「そもそもAIの構造推論に頼らずreconcileが機械強制する」だった。ポインタ案も結局「AIに実コードを読ませて構造推論させる」という同じ弱点を残すため、論点1の結論（reconcile拡張）で代替される。
**根拠:**
- ポインタであっても「見て真似る」という運用である限り、コンテキスト次第で抜け漏れが起きるという今回の事象の再発を防げない
- 一方reconcile（uc-lint-docstring）はAIの読解を介さず違反を機械的に列挙するため、書く前にも書いた後にも同じ基準で強制できる

### 合意決定
**決定:** 論点1の結論に統合。CodingSchemaのexampleフィールド（ポインタ含む）は追加しない。
**理由:** reconcile拡張の方が「言語で書き方が決まっている不変ルール」を機械的に担保するという目的に直接的に応える。
**次のアクション:** なし（論点1のアクションに吸収）。

## 論点 3: SkillSchemaのStepsに「実行例」を追加する価値はInvocationSpecの実例パターンと同じ理屈で正当化できるか

### AI 初期見解
**見解:** 部分的に正当化できるが、InvocationSpecの実例（cliExamples/mcpExamples）と同じ種類の例を重複して足すのは無駄。Stepsに足すなら「どう呼ぶか」ではなく「AIがどう判断するか」の実例であるべき。
**根拠:**
- InvocationSpecの実例は「engineをどう機械的に呼ぶか」（構造化パラメータの実例）を示す。Stepsの弱さは別の層の話で、「AIが自然文の依頼をどう解釈してどのstepを選ぶか」という判断の実例が無いこと
- 判断の実例が無いと、AIはsummary+bulletsという抽象的な記述から毎回ゼロ推論することになり、既存のInvocationSpecの実例パターンが解消した問題（呼び出しの曖昧さ）と対になる問題（手順選択の曖昧さ）が残ったまま
- ただし全stepに実行例を義務化すると冗長になりやすい。既存4 engine skillの実績を見てから、本当に判断が割れやすいstepにだけ絞って追加する方が実効的

## 論点 4: 「弱さ」の正体は本当にexample不足なのか、それとも別の軸（粒度のばらつき・test-standardだけ突出した非対称性）なのか

### AI 初期見解
**見解:** 両方が同時に真である。example不足は本物の設計ギャップだが、test-standardとの厚みの差は設計欠陥ではなく「使い込まれた量の差」であり、schemaを直しても自動的には埋まらない。
**根拠:**
- test-standardが厚いのは、今日だけでも「テストファイル命名規約」「配置層の機械対応」「spec語彙の閉じ込め」等、実際に痛みを伴って発見された決定が`rules`ブロックに蓄積された結果であり、schema設計の産物ではなく運用の産物
- coding-standard/architectureは同じ密度の意思決定をまだ経験していない（例えば「型注釈を公開関数に付ける」等の一般論に留まり、waffle固有の失敗から得た教訓がまだ乗っていない）
- よって対策は二段構え: (a) example不足はschema拡張で埋める、(b) 厚みの非対称性はschemaを直しても消えず、今後coding-standard/architectureに関する実際の判断が発生するたびに`rules`へ追記する運用を続けることでしか埋まらない
