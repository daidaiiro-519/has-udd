# ブレインストーミング: DDDエキスパートアドバイザー knowledgeSkill 設計

**目的:** 「ドメイン駆動設計をはじめよう」PDFの内容を元に、ユーザーの質問・相談に対してDDDの観点からアドバイスを返す knowledgeSkill の設計を決定する
**モード:** アイデア発散

---

## アイデアダンプ

knowledgeSkill の設計候補として考えられるアプローチ：

1. **概念辞書型** — DDDの用語（ドメイン・サブドメイン・境界づけられたコンテキスト等）を定義辞書として持ち、質問された用語を引いて解説する
2. **質問パターン型** — よくある質問（「境界をどこで引くか」「集約の設計は？」等）をパターン化し、それにマッピングして回答する
3. **判断フレームワーク型** — DDDの意思決定フロー（「これはコアか汎用か？」「集約か値オブジェクトか？」）をフローチャートとして持ち、ユーザーの状況に応じて診断する
4. **章構造型** — 本の章ごとに知識をまとめ、関連章を参照しながら回答する
5. **レイヤー横断型** — 戦略設計・戦術設計・実装パターンの3層で知識を整理し、質問の抽象度に応じてレイヤーを選んで回答する
6. **具体例リスト型** — Uber・Google・スターバックス等の実例をセットで持ち、概念説明に例を必ずセットで返す
7. **アンチパターン付き型** — 正しいパターンとアンチパターンをペアで持ち、「やってはいけないこと」まで含めて回答する
8. **コンテキスト診断型** — ユーザーが自社の状況（業種・チーム規模・システム規模）を入力すると、DDDの適用可否・どの概念を使うべきかを診断する
9. **ステップガイド型** — 「DDDをどこから始めるか」のロードマップを持ち、現在地を聞いて次のアクションを返す
10. **ハイブリッド型** — 概念説明・判断支援・具体例・アンチパターンをセットにした複合フォーマット

**絞り込み候補:**

| 候補 | 実現性 | インパクト | 選定理由 |
|---|---|---|---|
| **ハイブリッド型** | 中 | 高 | 概念・判断・例をセットで返すと実務で使いやすい。これをベースに設計する |
| **判断フレームワーク型** | 高 | 高 | DDD最大の難所は「どこで境界を引くか」の判断。ここを強化すると差別化できる |
| **アンチパターン付き型** | 高 | 高 | 初学者が詰まるのは「何が間違いか」の理解。正/誤ペアは必須 |

## 論点 1: knowledgeSkill の SKILL.md 構造はどう設計すべきか

### AI 初期見解

**見解:** SKILL.mdは「インプット分類 → knowledge参照 → 出力フォーマット指定」の3ステップ構造にすべき。knowledgeファイルは別ファイルに分離して参照させる設計が最も拡張しやすい。

**根拠:**
- DDDは概念数が多く（20以上）、SKILL.md内に全知識を詰め込むと肥大化してメンテ不能になるから
- 本のPDF読み込みを段階的に進める予定のため、knowledgeファイルを増やしていける構造が現実的だから
- 既存のbrainstormスキルがtemplate参照モデルを採用しており、プロジェクトの一貫性がとれるから

### ユーザー見解

> ✏️ _（あなたの考え・反論・追加情報をここに書いてください）_

### AI 再考見解

**見解:** SKILL.mdは「索引＋ルーター」として設計する。責務は「Inputの解釈 → 参照すべきknowledgeファイルの特定 → 回答生成の指示」までに限定し、知識そのものは持たない。knowledgeファイルはセマンティックな粒度（=意味的にひとまとまりになる概念単位）で分割し、本の章立てをカテゴライズの基準として使う。

**根拠:**
- SKILL.mdを「索引」と捉えることで、Skillsのdescription参照と同じ仕組みで動作する設計になり一貫性がある
- knowledgeファイルのセマンティック粒度を担保することで、ファイルが増えても検索容易性が維持できる
- 章構造ベースのカテゴライズは本の内容と対応しており、PDFを段階的に読み進めながらファイルを追加していく作業フローと相性がよい

### 合意決定

**決定:** SKILL.mdは索引＋ルーターに徹し、「Input解釈 → knowledgeファイル特定 → 回答指示」の3責務のみを持つ。knowledgeファイルは本の章構造に沿ったカテゴリー分けで、概念単位のセマンティックな粒度で作成する。
**理由:** 責務分離により拡張性・検索容易性・メンテナンス性がすべて確保できるため
**次のアクション:** 論点2でknowledgeファイルの具体的なフォーマットを決定する

## 論点 2: knowledge ファイルのフォーマットはどう表現するか

### AI 初期見解

**見解:** 以下の構造を1概念1ファイルで持つべき。「定義 → いつ使うか → 判断基準 → 具体例 → アンチパターン → 関連概念」の順。

```markdown
## {{概念名}}

**定義:** 1〜2文の簡潔な定義

**いつ使うか:**
- 条件1
- 条件2

**判断基準（チェックリスト）:**
- [ ] 問い1: YESなら〜、NOなら〜
- [ ] 問い2: ...

**具体例:**
- 良い例: ...（本書の実例引用）
- 悪い例（アンチパターン）: ...

**関連概念:** [[概念A]] [[概念B]]
```

**根拠:**
- 「いつ使うか」と「判断基準」を分けることで、概念理解と意思決定支援の両方に使えるから
- 1概念1ファイルにすると、AIが必要な知識だけをピンポイントで参照でき、余分なトークンを消費しないから
- アンチパターンを必ずペアにすることで、初学者が陥りがちな誤りを予防できるから

### ユーザー見解

> 判断フローはテキストで十分。Mermaidは「人間がknowledgeファイルを追加・編集するときの地図」として concept-map.md に1つ持てばいい。人間はコンセプトマップから知りたい情報にたどり着けるのでいい感じ。

### AI 再考見解

**見解:** knowledgeファイルのフォーマットはMermaidを含まないテキスト統一で確定。concept-map.mdは「AIのルーティング用ではなく、人間のナビゲーション用」として別途持つ。この2つは読者が異なる索引として共存する。

**根拠:**
- AIはSKILL.mdの索引テーブルで十分にルーティングできるため、Mermaidのトークンコストは不要
- concept-map.mdは人間がknowledgeファイルを追加・拡張するときの設計地図として機能する
- 「knowledgeファイルが増えて把握しきれなくなったタイミング」で作ればよく、初期段階では不要

### 合意決定

**決定:** knowledgeファイルのフォーマットはMermaidなし。構成は「定義 → 分類（概念による） → 判断基準（テキストフロー） → 比較表 → 具体例 → アンチパターン → 関連概念」。concept-map.mdは人間用ナビゲーションとして別管理。
**理由:** AIと人間で索引の役割を分離することで、トークン効率と人間の可読性を両立できるから
**次のアクション:** 論点3でインプット受け取り方・アウトプットフォーマットを決定し、SKILL.mdの実装に進む

## 論点 3: ユーザーインプットの受け取り方・アウトプットのフォーマットはどうすべきか

### AI 初期見解

**見解:** インプットは「自由記述」のまま受け取り、Skillがインプットを3タイプに自動分類して応答フォーマットを切り替えるべき。

| 分類 | 例 | 応答フォーマット |
|---|---|---|
| **概念質問** | 「境界づけられたコンテキストとは？」 | 定義 + 具体例 + 関連概念 |
| **判断相談** | 「この機能はコアサブドメインですか？」 | 診断チェックリスト + 判定結果 + 理由 |
| **実装相談** | 「集約のルートはどう決める？」 | 判断基準 + コード例（あれば） + アンチパターン |

**根拠:**
- ユーザーに「質問タイプを選んでください」と聞かせると摩擦が生じるから、自動分類が体験として優れているから
- 同じ「知識」でも質問の目的によって最適な回答形式が異なるから
- このタイプ分類自体をSKILL.mdのStep 1として明示することで、Claudeの判断にブレがなくなるから

### ユーザー見解

> 実際の使用シミュレーションを通しで見たい。インプット→思考→アウトプットの流れで確認したい。また出力パターンはプレースフォルダー付きスケルトンテンプレートで管理し、プレースフォルダー説明票もセットで用意すること。

### AI 再考見解

**見解:** 3タイプの出力テンプレートをスケルトン化し、SKILL.mdにプレースフォルダー説明票を共通・タイプ別に分けて記載する。説明票に「knowledgeファイルのどのセクションから引用するか」まで明記することで再現性を担保する。

**根拠:**
- プレースフォルダー説明票がないとClaudeの出力にばらつきが生じ、テンプレートの意味がなくなるから
- 共通とタイプ別に分けることで、説明票の重複を避けつつ網羅性を確保できるから
- 「引用元セクションを明示する」ことでknowledgeファイルとテンプレートが疎結合になり、知識の更新が出力に自動反映されるから

### 合意決定

**決定:** 3タイプのスケルトンテンプレート（concept / judgment / implementation）をreferences/に配置し、SKILL.mdのStep 3に共通・タイプ別のプレースフォルダー説明票をセットで管理する。
**理由:** テンプレート＋説明票のセットが再現性の最小単位であり、保守性も確保されるから
**次のアクション:** PDFの残章を読み進め、knowledgeファイルを順次追加していく

## セッションまとめ

### 合意事項

| 論点 | 合意内容 |
|---|---|
| SKILL.mdの構造 | 索引＋ルーターに徹する。責務は「Input解釈 → knowledgeファイル特定 → 回答指示」の3つのみ |
