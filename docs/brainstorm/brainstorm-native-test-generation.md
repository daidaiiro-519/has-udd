# ブレインストーミング: TestScenariosを.feature/BDDランナー経由ではなく、ネイティブテストとして生成する

**目的:** spec由来の`TestScenarios`を実行可能にする際、Gherkin(`.feature`)とBDD専用ランナー(behave等)を経由する現行方針を見直し、各言語のネイティブなテストフレームワークへ直接renderする設計に切り替える。
**モード:** 設計判断（論点 → 見解 → 合意）
**関連:** `brainstorm-spec-feature-execution-binding.md`（本ブレストにより実質的に上書き・スーパーシード）。SP-4/SP-5（design-spec-schema）の「シナリオは1回だけ定義し、ステップバインディングで観点を切り替える」という核となる原則は維持する。

---

## 発端

spec由来の`.feature`ファイルが、これまで一度も`behave`に実行されていなかったという欠陥（`brainstorm-spec-feature-execution-binding.md`）の対処を検討する過程で、「そもそもBDD専用ランナーに依存すること自体が正しいか」という、より上位の疑問が生じた。

## 論点1: BDD専用ランナー vs 各言語ネイティブなテストフレームワーク

### 調査結果（8スタックで実施・全て検証済み）

| 言語 | BDDツール | 発見メカニズム |
|---|---|---|
| Python | behave | `features/`自動再帰探索・設定不要 |
| Python | pytest-bdd | 明示設定必須（`bdd_features_base_dir`） |
| JS/TS | Cucumber.js | `features/`自動再帰探索 |
| Java | Cucumber-JVM | ランナークラスと同じパッケージ（固定フォルダなし）・明示設定必須 |
| Go | godog | `features/`自動再帰探索 |
| Rust | cucumber-rs | `tests/features/` |
| C# | Reqnroll | `Features/`（自動発見） |
| C++ | cucumber-cpp / cwt-cucumber | wireプロトコル方式（別プロセス）／モダン実装は採用実績小 |

**発見:** 「Gherkinで書けばどの言語でも動く」という前提は誤り。BDD専用ランナーは言語ごとに競合する複数の選択肢があり（例: Python=behave vs pytest-bdd、C++=cucumber-cpp vs cwt-cucumber）、フォルダ規約も発見メカニズムもバラバラ。C++に至っては業界標準と呼べるツールが無い。

### AI見解

**見解:** `.feature`ファイル（Gherkin構文）を生成物として作ることをやめ、`TestScenarios`から各言語の**ネイティブなテスト関数**（pytestの`def test_...`・Vitestの`test(...)`・JUnitの`@Test`等）を直接render_engineが生成する方式に切り替える。

**根拠:**
- ネイティブテストフレームワーク（pytest/Vitest/JUnit/go test/cargo test/xUnit）は、各言語で実質一択の、BDD専用ランナーよりずっと成熟した標準。C++のような「定番が無い」状況を回避できる。
- ただしフォルダ・命名規約自体のばらつきはネイティブテストでも残る（Goはソースと同居する`_test.go`、Rustは`tests/`等）。これは解決しない。
- 変わるのは「ツール選択の余地」がほぼ消えること（BDDランナー選びという二重の不確実性を無くせる）。

**新たに生じるコスト（隠さず明記）:** render_engineは今「Gherkinテキスト生成ロジック1つ」で全言語に対応できていた（Gherkin構文自体が言語非依存のため）。ネイティブテスト方式では、**render_engineが各言語のテスト関数構文ごとにテンプレートを持つ必要がある**。フラグメンテーションは減るが、render_engine側の実装コストは増える。

### PoCで検証済み（`/tmp/waffle_bdd_poc/`・未コミット）

実際の`uc-query-document.json`のTestScenarios（4シナリオ）を使い、以下を実証:
1. TestScenariosから、pytestのネイティブなテスト関数を直接生成（Gherkin/`.feature`を経由しない）
2. 生成されたテスト関数が、実際の`QueryEngine`を呼び出すステップ実装関数を経由して実行され、**4件全てpass**
3. ステップ実装を1つ欠けさせると、Gherkinの「undefined step」的な曖昧な扱い（behaveはデフォルト非strictだと警告のみで失敗にならない）ではなく、**`AttributeError`として即座に失敗する**ことを確認——静かにスキップされない、より安全な失敗モード

### ユーザー見解
> 合意。JS/TSはJestよりVitestの方が好み。

### 合意した内容
`.feature`/BDD専用ランナーをやめ、各言語ネイティブなテストフレームワークへ直接renderする方式を採用する。JS/TSの参照実装はVitestを標準とする（API自体はJestとほぼ同一の`test(...)`構文）。
**なぜこれに合意したか:** 8スタックの調査でBDD専用ランナーのフラグメンテーション（複数競合ツール・C++は業界標準なし）が実証され、PoCでネイティブテスト方式が実際に動作することも確認できたため。
**次のステップ:** 論点2（TestScenariosの中身・SP-4/SP-5原則への影響）の検討に進む。

## 論点2: TestScenariosの中身・SP-4/SP-5原則への影響

### AI見解

**見解:** `TestScenarios`ブロックの中身（Given/When/Thenのプレーンテキスト・`category`/`viewpoint`/`covers`フィールド）は**一切変更しない**。変わるのはrender先の成果物形式だけ。

**根拠:**
- SP-4/SP-5の核心（「シナリオは1回だけ定義し、ステップバインディングで観点を切り替える」）は、Gherkin構文に依存した原則ではなく、「単一のシナリオ定義から複数の実行系にバインドできる」という抽象的な性質。ネイティブテスト生成でも、同じJSONから言語ごとに異なるテスト関数を生成できる限り、この原則は維持される。
- `covers`フィールド（シナリオが対応する受け入れ基準を明示）は、AIによる無自覚な重複シナリオ生成を抑止する既存の構造的歯止めとして機能する。

### ユーザー見解
> `.feature`ファイルは残し、render_engineが今まで通り機械的に生成する。ただし`.feature`はGherkinランナーに実行させる対象ではなく、**AIがネイティブテストコードを書く際に読む「実装すべき仕様書」**という位置づけに変える。stepsファイルへの分割は不要——AIが`.feature`を読んで、ネイティブテストファイルに直接assertロジックまで書けばよい。TDDで`.feature`を育てる感覚に近い。

### AI再考見解（合意）
**見解:** ユーザー案を採用。render_engineの責務は変更せず「`.feature`を機械的に生成するだけ」のまま維持する。ネイティブテストファイルはAIが`.feature`を読んで直接執筆する（生成スタブ＋別stepsファイルという分割は撤回）。

**根拠:**
- render_engineに言語別テンプレートを追加するコスト（論点1で懸念した新規コスト）が丸ごと消える。Gherkin生成ロジック1つのままで済む。
- `.feature`がGherkinランナーに一切実行されない「非実行の参照専用artifact」になるため、8スタック調査で発見したBDDランナーのフラグメンテーション問題がそもそも発生しない。
- stepsファイルの新規作成をrenderが担うかという設計上の難点（論点2直前で議論した「既存なら触らない・無ければ作る」という特殊分岐の要否）も、分割自体をやめることで消滅する。

**ドリフト検知の機械化（追加合意）:** `.feature`(≒TestScenarios)とAIが書いたネイティブテストファイルの乖離は、**シナリオ名とテスト関数名の集合演算（AST解析・実行不要）**で機械的に検出できる:
```
AcceptanceCriteria --(covers)--> TestScenarios --(名前突き合わせ)--> ネイティブテスト関数
```
- spec側にあってテスト側に無い名前 → 未実装のシナリオ
- テスト側にあってspec側に無い名前 → 孤立した古いテスト

**中身（assertロジックがシナリオの意図と本当に合っているか）までは機械検出しない**——検出できた差分（未実装/孤立）だけをAIが見て、中身の妥当性を評価する役割分担とする。全てを機械判定しようとせず、コストのかかる意味理解はAIに委ねる。

**合意した内容:** `.feature`は残し引き続きrender_engineが機械的に生成する。ただし非実行の参照専用artifactとし、Gherkinランナー（behave等）には一切依存しない。ネイティブテストファイルはAIが`.feature`を見て直接執筆する（生成スタブ＋stepsファイルの分割は撤回）。ドリフト検知は「シナリオ名⇔テスト関数名」の名前突き合わせで機械化し、検出後の中身評価はAIが担う。
**なぜこれに合意したか:** render_engineの実装コストを増やさずに済み、BDDランナーのフラグメンテーション問題も回避でき、TDDで「.featureを育てる」という自然な感覚にも合っているため。
**次のステップ:** 論点3（test-standardに何を追加するか）は、この新しい形（.featureのみ生成・ネイティブテストはAI執筆）に合わせて内容を見直す。

## 論点3: test-standard(CodingSchema)に何を追加するか

### AI見解

**見解:** `test-standard-{stack}.json`に以下のフィールドを追加する:
- `testFramework`: このスタックが使うネイティブテストフレームワーク名（例: `"pytest"`）
- `testScanRoot`: テストランナーが自動探索する（またはそのプロジェクトが採用する）フォルダパス
- `testDiscovery`: `"auto"` / `"co-located"`（Go/Rustのようにソースと同居）等の区別

render_engineは、対象usecaseの`x-render-target`を解決する際、**現在の`test-standard`ドキュメントを都度参照して**、これらの値を使い出力パスとテンプレート形式を決定する。

**根拠:** BDD専用ランナー時代に想定していた「featureScanRoot/featureDiscovery」という設計をそのまま流用できる（本質は変わらず、対象がBDDランナーからネイティブテストランナーに変わっただけ）。C++のように業界標準が定まらないスタックについては、Waffle側でデフォルト値を決め打ちせず、そのプロジェクトのtest-standard著者が明示的に埋める。

### ユーザー見解
> 論点2の合意（render_engineは`.feature`だけを生成・ネイティブテストはAIが直接執筆）に合わせて内容を見直すべき。

### AI再考見解（合意・実装済み）
**見解:** `x-render-target`側のフィールド追加（`testScanRoot`/`testDiscovery`をrender_engineが読む）は不要になった。代わりに、`test-standard-{stack}.json`自体の記述を、**AIがテストを書く際に読むガイダンス**として書き換える。

**実装した変更（`test-standard-python-hexagonal.json`）:**
- `testTypes`: `acceptance-bdd`（tool: behave）→ `acceptance`（tool: pytest・「AIが`.feature`を読んで直接執筆」）
- `framework.acceptance`: `behave` → `pytest`
- `scenarioBinding`: 「ステップ実装の配置」を削除し、「ネイティブテストの配置」（`tests/acceptance/test_{spec-id}.py`）に置き換え。「ドリフト検知」項目（シナリオ名⇔テスト関数名の名前突き合わせ）を追加
- `placementByTarget`: `usecase の受け入れ`のtestKindを`BDD`→`受け入れ（ネイティブ・.feature は参照専用）`、pathを`features/`→`tests/acceptance/`
- `rules`: 「TestScenarios は .feature で実行可能にする」→「.feature は実行対象ではなく、AIが読んで実装すべき参照専用の仕様書とする」（意味が反転）
- **CodingSchema/v2.json側**: `testType`のenum値`acceptance-bdd`を`acceptance`に改名（スキーマレベルの語彙変更）

**検証:** JSON妥当性・`waffle validate`（VALIDATED）・pytest 33件・behave 70シナリオ/345ステップ、全て緑を確認済み。

**合意した内容:** test-standardは「render_engine向けのパス解決設定」ではなく「AIがネイティブテストを書く際に従うガイダンス」として位置づけ、上記の通り実装した。
**なぜこれに合意したか:** 論点2でrender_engineの責務を「`.feature`生成のみ」に確定させたため、test-standard側の役割もそれに合わせて変わる必然性があった。
**次のステップ:** 論点4は既に合意済み。本ブレストの4論点全てが合意に至った。

## 論点4: AIによる無駄テスト・重複テストの防止効果

### AI見解

**見解:** **大幅に軽減されるが、完全な防止は保証されない。**

**根拠（防げる理由）:**
- 生成されるネイティブテストファイルは100%自動生成・毎回全体を`TestScenarios`から上書き（差分パッチではない）。AIが生成フォルダへ直接テスト関数を書き足しても、次のrenderで消える。
- これにより「テストを増やす」という行為は、構造的に「spec側のTestScenariosにレビュー可能な1エントリを追加する」ことへ誘導される。
- `covers`フィールドが既に「このシナリオが何をカバーするか」の明示を強制しており、内容の重複が起きにくい設計になっている。

**根拠（防げない理由）:**
- TestScenarios自体に内容が近い重複シナリオが追加されることは、技術的には防げない（レビュー運用に依存）。
- Waffleのワークフローを経由せず、AIが生の`test_*.py`をリポジトリのどこかに直接書く行為そのものは、物理的に阻止できない（規律運用の話）。

### ユーザー見解
> シナリオでカバレッジを満たせるようなシナリオを用意することの方が大事。Waffleの開発はTDDで進めるため、コードは常にシナリオに従って書かれる。シナリオ自体がソースコードの品質と仕様の両方を左右するので、コードカバレッジ計測やusecase↔ソースファイルの対応付け（C-2）より優先度が高い。

### AI再考見解（合意）
**見解:** コード行カバレッジ%・usecase↔ソースファイルの対応付け（C-2）は二次的な話であり、Waffleにとって一次的な「カバレッジ」は**シナリオそのものの完成度**である。

**根拠:**
- TDDで進める以上、コードは常に先に書かれたシナリオ（テスト）に従って書かれる。ディシプリンが守られていれば、シナリオが要求しないコードは存在しない。よってシナリオ集合が十分であれば、コードカバレッジは事後測定するまでもなく結果として満たされる。
- 逆に言えば、コードカバレッジ%を先に測っても、それはシナリオが既に書いた分の「後追いの確認」にしかならず、TDDのワークフローにおいて能動的な改善レバーにはならない。
- したがって能動的なレバーは、シナリオ側の完成度を機械的にチェックする2点に絞られる:
  1. `AcceptanceCriteria`のうち`covers`されていないものが無いか（未カバー検出）
  2. 各usecaseのシナリオが`category`（正常系/境界値/異常系）の観点で偏っていないか

**合意した内容:** コードカバレッジ計測・usecase↔ソースファイル対応付け（C-2）は本ブレストの対象外とし後回しにする。「シナリオレベルのカバレッジ」（AcceptanceCriteria未カバー検出＋category偏り検出）を、将来のquery engine拡張候補としてバックログに記録する。
**なぜこれに合意したか:** WaffleはTDD前提で開発されるため、シナリオの完成度が全ての起点であり、コードカバレッジという事後指標より優先度が高いというユーザーの指摘は、Waffleの開発プロセスの実態に即している。
**次のステップ:** 「AcceptanceCriteria未カバー検出」「categoryの偏り検出」を、query engineの新規operationまたは新しいreconcile系usecaseの候補として、ロードマップのバックログ（Stage B関連）に追記する。

**追記（コード行カバレッジ(C0/C1)の位置づけ・合意）:**
コード行カバレッジ(C0)・分岐カバレッジ(C1)を**目標として追いかけることには意味が無い**（カバレッジを稼ぐためだけの無意味なテストを誘発する、既知のアンチパターン）。しかし**診断ツールとしてなら、シナリオ側の完成度チェックと異なる方向の価値が1つだけある**:

- シナリオ側チェック（AcceptanceCriteria未カバー検出）＝**上から下**（要求→シナリオ）の確認
- コードカバレッジ＝**下から上**（実装コード→どこが叩かれていないか）の確認、逆方向で補完関係

「どのシナリオからも実行されない行・分岐がある」ことが分かった場合、原因は2つに絞られる: (a) 起こり得ないケースへの不要な防御コード（削除すべきサイン）、(b) 誰も気づいていなかったシナリオの書き漏らし。TDD規律が完璧なら理論上不要だが、**規律からのドリフトを検出する後付け監査**としてのみ、将来検討の余地を残す（目標にはしない）。
