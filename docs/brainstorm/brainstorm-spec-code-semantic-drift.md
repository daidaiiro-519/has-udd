# ブレインストーミング: 仕様(spec)プロースと実装コードの意味的driftを防ぐ手段

**目的:** 今回発見した「specの文章内容(prose)が実装の実際の挙動と乖離している」ドリフト(意味的drift)を、機械的・運用的にどこまで再発防止できるか、既存の4つのreconcileチェックの守備範囲を整理した上で決める。
**モード:** 問題解決

---

## 問題の再定義

**spec proseの意味的drift(内容の正しさ)を検知する機械的な手段が無いため、人手(AI監査)でしか気づけず、放置すると仕様が実装から乖離したまま陳腐化する。**

今回の具体例(1セッションで発見した8件):
- uc-scaffold-document: `SKIPPED`をErrorsのコードとして誤記載(実際は成功パスのリストキー)
- uc-query-document: 「常に`{prompt, value}`」という断定が、schemaRef無しファイルのscan以外の操作では`{type, content}`という別形状になる例外を見落としていた
- uc-validate-document: `VALIDATION_FAILED`というコードを主張していたが実装は`details`=違反リストのみでコードを返さない
- uc-scan-source-code / uc-lint-docstring: google以外の4kind(tsdoc/javadoc/godoc/rustdoc)が現状動作するかのような記述だったが実際はadapter未実装でUNSUPPORTED_KINDになる
- uc-check-spec-integrity: mainFlowの「usecase間の相互参照」という不正確な文言、`entities[0]`のみを見る潜在バグ
- agg-document: `render`の状態ゲート(未検証ではrenderできない)・`activate`/`deprecate`コマンドは実装が存在しない宣言だった
- agg-schema: 撤去済みの`publishVersion`コマンドへの言及が残っていた

これらはいずれも**機械的（構造・参照整合性の集合演算）には検出できず、AIが並列エージェント監査でproseを1行ずつ読んで初めて発見**したもの。

再発防止として`@spec`アンカー(DocCommentにspec IDをタグ付けし逆引き投影を作る設計)を一度実装したが、これは`brainstorm-ai-era-detail-design.md`→`brainstorm-coding-schema.md`で「リンクは管理でなく計算するもの。正典配置(パス)＋検索(code_scan)で代替」として明確に撤回済みの設計だったため、ユーザー指摘によりrevertした。今回のブレストは**アンカー方式を使わない前提**で進める。

## 原因候補

### ①現状の仕組みで防げているドリフトの整理（洗い出し）

現在稼働している機械的drift検知は4種類。いずれも**構造・参照が「実在するか」の集合演算**であり、prose本文の意味は読まない。

| # | usecase | 検知対象 | 手法 |
|---|---|---|---|
| 1 | uc-check-schema-version-drift | Documentが参照するschemaRefの実在性・最新版・`missing_declared_fields`(参照先schemaが宣言する値フィールドをDocumentが実際に持つか) | 文字列比較・版番号抽出・fillTemplateパスの存在確認 |
| 2 | uc-check-scenario-drift | spec宣言のシナリオ名⇔テスト関数名の対応、gherkin本文⇔テストdocstringの内容一致 | 命名規則変換(sanitize)＋行単位の連続部分列マッチ(`contains_subsequence`) |
| 3 | uc-check-spec-integrity | bc.json/subdomain.json宣言 vs ディスク実態の10項目(孤立値オブジェクト・undeclared_document_fields・subdomainRef/aggregateRef不一致等) | 集合の差分演算(宣言集合 vs 実在集合) |
| 4 | uc-check-agent-skill-drift | subagentのskillPreloadsが参照するSkillの実在性・プリロード可能性(manualOnly制約) | ファイル存在確認＋フィールド値の突合 |

これらは全て「名前・パス・参照キーが存在し、整合しているか」という**構造的drift**であり、今回発見した8件は1件もこの4つでは検知できなかった（scenario-driftの`gherkin_mismatches`だけが唯一「内容」に触れるが、それも**行単位の文字列一致**であって意味理解ではない——実際、今回の副次発見（uc-query-documentのGiven行の文言差異）はこれで検知できたが、それは表現の一致/不一致の話であり、「主張している挙動が正しいか」とは別種）。

### ②今回発見したような意味的drift(prose⇔実装)のうち、何が防げていないか（表面的原因）

- mainFlow/postconditions/acceptanceCriteria/errors/invariantsという**自然言語プローズの「意味」**は、決定的な文字列/集合演算だけでは検証できない
- 「常に」「必ず」等の断定表現が例外ケースを持つかどうかは、実装の分岐ロジックを読んで判断するしかない
- 未実装の設計（将来ロードマップ）を実装済みであるかのように書いてしまう誤りは、prose単体からは判別できない(実装ファイルとの突合が要る)
- 集約(agg-document等)が宣言する不変条件と、それを実際に守るはずのusecase engineとの対応関係(どのengineが何を守るか)が、宣言だけでは検証できない

### 根本原因

- **技術的限界**: 自然言語プローズの意味理解には、決定的な文字列/集合演算ではなくAIの意味理解が必要。8件中6件（uc-query-documentの断定漏れ・uc-scan-source-code/uc-lint-docstringの未実装スコープ記述・uc-check-spec-integrityのmainFlow不正確な文言・agg-documentのrender状態ゲート/activate・deprecate・agg-schemaのpublishVersion言及）はこの種類で機械検出不能。**残り2件（uc-scaffold-documentのSKIPPED・uc-validate-documentのVALIDATION_FAILED）は「宣言されたエラーコードが実装ファイルに存在するか」という機械チェックで実際に検出可能**（今回試作したcheck-error-code-driftのロジック自体は機能していた。撤回理由はリンク機構がアンカー方式であったことへの設計原則違反であり、チェックロジックの誤りではない）
- **発火点が無い**: spec変更時・実装変更時に自動的にreconcileが走る仕組み(Hooks)が未実装（`design-hooks.md` Phase5・H-3/H-5は「@spec逆引き」を前提に書かれており、アンカー全廃後に**更新されないまま放置されている**）。今回は「ユーザーに聞かれたから」全spec相手に手動監査した
- **正典配置(canonical path)の変換規則が明文化されていない**: 実際に確認したところ、既存10usecase全てが「`uc-`プレフィックスを外し、末尾の`-document`があれば外し、ハイフンをアンダースコアに変え、`_engine.py`を付ける」という**単一の決定的変換で100%一致**していた（例: `uc-scaffold-document`→`scaffold_engine.py`）。パス規則自体は不規則ではなく、むしろ完全に規則的——ただし`test-standard-python-hexagonal.md`にはテストファイル名規約（`test_{documentId}.py`）は明文化されているのに、この**実装engineファイル名規約は明文化されていない**。明文化されていない規則は将来のusecase追加時に守られる保証が無い

## 論点 1: 意味的driftの検知は、どこまで機械化でき、どこからAI判断に委ねるべきか

### AI 初期見解
**見解:** 完全自動化はできないが、「宣言された値(エラーコード等)が実装ファイル内に文字列として存在するか」という狭い部分集合は機械化可能で、実際に8件中2件（1/4）がこの部分集合に該当した。一方でmainFlow/postconditions/invariantsのような自然言語の意味理解（断定の例外・未実装スコープの記述・状態ゲートの実装漏れ）は、現状の技術（正規表現・集合演算）では原理的に不可能で、8件中6件（3/4）を占める。この領域はAIによる定期監査（reconcileの一形態）として運用するしかない。

**根拠:**
- 今回発見した8件のうち、2件（SKIPPED/VALIDATION_FAILED）は「宣言コード⇔実装ファイル内の文字列存在」という機械チェックで検出可能だった（試作済み・ロジックとしては機能を確認済み）
- 残り6件（断定の例外見落とし・未実装機能の記述・状態ゲート実装漏れ・撤去済み機能への言及）は、いずれも自然言語文の解釈と実装コードの読解を要する（構文的パターンマッチでは判別不能）
- 正典配置(canonical path)自体は10/10で規則的な変換が成立することを確認済み（論点2参照）——機械化の障害は「パスの不規則性」ではなく「意味理解の必要性」にある

## 論点 2: 「宣言された値が実装に実在するか」的な部分機械化を、アンカー無しでどう実現するか

### AI 初期見解
**見解:** ★訂正: 当初「正典配置は不規則だから1:Nの緩い一致に条件を緩めるべき」と考えたが、実際には既存10usecase全てが「`uc-`プレフィックスを外し、末尾の`-document`があれば外し、ハイフンをアンダースコアに変え、`_engine.py`を付ける」という**単一の決定的変換で100%一致**していた（曖昧な緩和は不要）。取るべき手段は、この変換規則を**coding-standard（test-standardと対になる場所）に正式な規約として明文化**し、それを使った1:1のパス導出関数を実装すること。アンカー全廃の思想（「リンクは管理でなく計算するもの」）に最も忠実な形。

**根拠:**
- 実測で10/10のusecaseが同一の変換規則に従っていた（`uc-scaffold-document`→`scaffold_engine.py`等）
- 現状test-standardにはテストファイル名規約（`test_{documentId}.py`）は明文化されているのに、実装engineファイル名規約が明文化されていないのは非対称。明文化すればscaffold等のツールが規約違反を機械検知できるようになる
- 1:1の決定的変換なら、緩い1:Nディレクトリgrepで生じる偽陰性（無関係なファイルの文字列と偶然一致）のリスクが原理的に発生しない

## 論点 3: AI判断が要る意味的drift監査を、いつ・どう発火させる運用にするか

### AI 初期見解
**見解:** 今のところ「ユーザーに聞かれたときに全spec手動監査する」以外の発火点が無い。`design-hooks.md`(Phase5)がこの発火点の受け皿として設計途中だが、H-3/H-5は「@spec逆引き」を前提に書かれており**アンカー全廃後に更新されないまま放置**されている。全spec相手に毎回今回のような並列AIエージェント監査をやるのはトークンコスト的に非現実的なので、「今回変更されたspec.json／今回変更されたapplication/usecases配下」だけを対象にする**差分ベースの監査**が現実的な落としどころではないか。

**根拠:**
- 今回は10個のusecase/aggregateを並列AIエージェントで監査した。spec数が増えるほどコストが線形以上に増える
- git diffで「変更されたspec.json」「変更されたPythonファイル」を特定できれば、その周辺だけ監査すれば済む
- `design-hooks.md`のH-3/H-5は本来この発火点を担う設計だったが、アンカー全廃という上位決定を反映できておらず、内容自体の更新が必要

## 現在地点(セッション引き継ぎメモ)

論点1〜3のブレスト後、実際に「実装にしか存在せずspecに無い」方向(方向A)の検知手段として、Hooksベースの具体設計まで話が進んだ。以下、次セッションが拾える状態にする。

### 合意済みの設計(論点3の実質的な発展形)

- **方向Aが主眼**: 「specに無いのに実装がある」は候補提示ではなく**それ自体が是正すべき違反**（フォルダ宣言vs実ファイルの集合差分という確定的事実であり、意味理解が要る他のドリフトとは性質が違う→**ブロックしてよい**）
- **判定軸は「新規ファイルか」ではなく「未宣言の概念名(クラス名)がそのファイルに現れたか」**（既存ファイルへの追記も対象。「ファイルは育つ」というユーザー指摘で訂正済み）
- **フック方式**: `PreToolUse`、`matcher: "Write|Edit"`。スコープ判定(対象パスが宣言済みドメインフォルダ=`ConceptPlacement`宣言下か)・結果内容の先読み(`Write`は`tool_input.content`そのまま、`Edit`は`old_string`→`new_string`をスクリプト側で適用して再現)は**Pre単体で完結**、Post不要
- **判定ロジックはトークン非消費**: 純粋スクリプト(AST走査+spec宣言リストとの集合差分)。AI呼出し(`prompt`/`agent`型フック)は使わない
- **2段構え**: Pre(即時ブロック) + commit時ゲート(`git pre-commit`、バックストップ。Preが機能していれば通常無言)。`Stop`フックは三重になるため不採用(YAGNI)
- **ツール3種(Claude Code/GitHub Copilot/Kiro)とも`PreToolUse`相当＋`exit 2`でブロックという核が共通**。判定ロジック本体は1本のスクリプトにし、各ツールの設定ファイルは薄い配線のみにすれば移植可能
