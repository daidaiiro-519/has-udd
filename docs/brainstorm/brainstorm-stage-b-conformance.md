**目的:** HOW品質を守る層のうち中核（code↔spec の整合・reconcile・投影）を設計判断する。特に OQ-3（ドメインモデル整合ゲート）を起点に、「コードが domain-model spec を実現しているかをどう検査するか」「アンカー粒度と多重度」「spec を持たないコードの線引き」「整合チェックの強制責務」を決める。
**モード:** 設計判断（論点 → 見解 → 合意）

> ⏸️ **保留（順序の見直し・ユーザー指摘）:** 整合ゲートは「conformant とは何か」＝**CodingSchema の code-template 規約＋動く最小サンプル＋効果測定**が確定して初めて作れる。よって本ブレスト（ゲート）は**後段**へ送る。**先に CodingSchema/code-template 確定 → 最小動作サンプル → 効果測定**。C-1（集約を持つか）・C-2（アンカー粒度）・C-3（spec無しコードの線引き）は実は **code-template 規約の決定**なので、そちらのブレストに吸収する。残るゲート固有は **C-4（強制・reconcile×Hooks）** で、これが本当の後段。

## 論点 C-1: ドメインモデル整合ゲート — どう検査するか／Document 集約は要るか

### AI 初期見解
**見解:** 整合検査は**3段に分ける**。(a)**構造整合**（命名・存在・シグネチャ・レイヤ配置）＝規約＋投影で**機械検査**、(b)**振る舞い整合**＝`.feature`/unit テスト、(c)**設計整合**（集約境界が妥当か・不変条件が本当にカプセル化されているか）＝**レビュー＋同じ言葉、機械では完全には無理**。
そして「**Document 集約オブジェクトは実コードに要るか**」は **YES（要る）**。現状の `application/usecases` は JSON dict を手続きで触っており、dm-document が言う Document 集約（commands: create/validate/render/supersede・invariants・lifecycle）を**実体化していない** → OQ-3 ゲートでは「未実現」と判定される。集約を実コードに導入して初めて「設計がコードに宿る」（DDD）し、整合ゲートが意味を持つ。
**根拠:**
- DDD: ドメインモデル＝POCO、ソースが同じ言葉を語る。集約＝不変条件のカプセル化で自由度を下げる中核。dict 手続きはこれを満たさない（貧血モデルの極＝本書アンチパターン）。
- 機械検査には限界（用語集は振る舞い・設計妥当性を表現できない）＝完全自動化は不可。3段の責務分けが現実的。
- Stage A 波及は避けられないが、移行レディ（gen-gap マーカー・アンカー先置き）で**集約導入を局所リファクタに封じ込められる**。

## 論点 C-2: アンカーの粒度と多重度・投影スキーマの形

### AI 初期見解
**見解:** アンカーは**「意味のある実装単位」（集約クラス・usecase サービス・アダプタ）に1つ**を基本にし、**多重度（many-to-many）は許容してデータで表現**する。粒度は「全関数」でなく実装エントリ単位。投影は**双方向**に持つ: `spec → [impl refs]` と `impl → [spec refs]`。
**根拠:**
- ヘキサゴナルでは1 usecase が inbound/application/outbound に跨る＝**多コードが正常**。1:1 を強制すると嘘になる（嘘のアンカー＝drift）。
- 投影は読み取りモデル＝**クエリの形**で持てばよい。多重度はスキーマで素直に表現し、強制しない。
- 双方向にすると「この spec の実装一覧」（カバレッジ）と「このコードが負う spec 一覧」（影響範囲）の両方が引ける＝保守と重複防止の両方に効く。

## 論点 C-3: spec を持たないコードの線引き

### AI 初期見解
**見解:** **「ドメイン/usecase を実現するコードだけが `@spec` 必須。純技術（infra/glue/util/shared）は `@stack` のみで `@spec` 不要」**。reconcile は `@spec` 必須の対象を **domain 層（集約/業務サービス）と application 層（usecase）に限定**し、adapters/shared は **`@stack` カバレッジだけ**見る。
**根拠:**
- DDD のレイヤ: ドメイン層/アプリ層＝WHAT（spec）が在る。インフラ層＝技術的関心事で spec を持たない（能力に `@stack` で紐付くだけ）。
- これで**偽 orphan を防ぐ**（`shared/result` に `@spec` を求めない）。
- 「何が spec 必須か」を**レイヤ＝パス規約で機械判定**できる → code-template 規約に乗る（`src/.../domain|application` は `@spec` 必須、`adapters|shared` は `@stack` のみ、等）。

## 論点 C-4: 整合/重複防止チェックを誰がいつ強制するか（reconcile engine と Hooks の責務境界）

### AI 初期見解
**見解:** **検出と強制を分離**する。**reconcile（投影＋差分）＝engine（純粋・副作用なし）**が「orphan / 未カバー spec / 規約違反 / 多重度」を**構造化レポート**で返す。**Hooks がそれを検証ゲート（commit / CI / render 前）で BLOCK** する。AI の「着手前に投影を引く（重複回避）」は機械ゲートにしづらい（生成“前”の行為）ので、**Orchestrator のガードレール／ワークフロー**で促す。
**根拠:**
- 責務分離: 検出（engine＝決定的関数）と 強制（Hooks＝ゲート/副作用）は別レイヤ。`design-hooks` の H 群に素直に乗る。
- ES-3（reconcile の帰属）は「独立 engine か query 拡張か」だが、**投影＝query の親戚**（読み取り）なので **query 拡張が有力**（別途 ES-3 で確定）。
- 重複防止の「着手前参照」は BLOCK 型ゲートに不向き＝**事前ガードレール**（Role/engine routing が「実装前に投影を引け」を指示）で担保するのが筋。
